## Main Thread: Moving Away from MCP Servers

David (DevOps / enterprise) and the philosophy behind the Pi coding agent converged on the same point: MCP servers load all of their tool descriptions into context every session — e.g. Playwright MCP ≈ 13.7k tokens, Chrome DevTools MCP ≈ 18k — which is roughly 7–9% of the context window consumed before any work begins, much of it for tools you won't use that session.

Preferred pattern instead: **a script or binary plus a skill that documents it**, invoked via bash. Pi takes this further — rather than installing a tool, you ask the agent to extend itself by writing the tool it needs. Several attendees also called out **project-specific skills** as a current favorite.

## Stacks

- **David** — OpenCode + Kimi via Fireworks inference. OpenSpec for spec-driven work (the only tool he's found that handles brownfield, not just greenfield). Uses Matt Pocock's skills repo. Hard "no" on MCP.
- **Nathan** — Among the first employees at Monogram (stealth iOS + Node backend startup, ~$40M raised). Runs Claude Desktop and Codex side by side; uses the Figma MCP daily (inconsistent reliability). Raised the context-switching burnout problem.
- **Jesse** — Optioneer at a ~4,000-person enterprise; constrained to approved tools (Copilot CLI plus an internal agent-team framework). Uses Joycraft on both his jobs. Dropped Cursor.
- **Max** — Claude Code in cmux. Keeping the stack deliberately simple while going deep on Pi and open-weight models.

## Open Problems Discussed

- **The waiting gap** — Agent runs take 1–10 minutes. Options: parallel agents on git worktrees (only safe with good specs), or monotask and step away. General agreement that running many parallel agents across projects causes real cognitive burnout — the human is still the context engine.
- **RAG is unreliable** for large or multi-repo context, and no one has a working answer. Karpathy's LLM Wiki was floated as an alternative: works well under ~100k tokens but breaks down on large vaults (e.g. David's ~2,000-note Obsidian).
- **Subsidy risk** — Concern that current subscription pricing is unsustainable, motivating interest in running open-weight models locally. Local inference is memory-bandwidth-bound, so large models need a lot of fast RAM (or a multi-Mac cluster).

## Tools & Models Mentioned

### Agents & CLIs

- **Pi** — Minimal terminal coding agent; anti-MCP; self-extending (the agent writes its own tools).
  [pi.dev](https://pi.dev/) · [write-up](https://mariozechner.at/posts/2025-11-30-pi-coding-agent/) · [Armin Ronacher's take](https://lucumr.pocoo.org/2026/1/31/pi/)
- **OpenCode** — Terminal coding agent, multi-provider. [opencode.ai](https://opencode.ai/docs/providers/)
- **Claude Code / Claude Desktop / Codex** — Referenced as the mainstream stack.
- **Copilot CLI** — Jesse's enterprise-approved tool.

### Spec / Workflow

- **OpenSpec** — Spec-driven development with brownfield delta specs (ADDED / MODIFIED / REMOVED markers); specs live next to the code as documentation. [github.com/Fission-AI/OpenSpec](https://github.com/Fission-AI/OpenSpec)
- **Joycraft** — Max's AI-harness scaffolder (in use by Jesse) [github.com/maksutovic/joycraft](https://github.com/maksutovic/joycraft)

### Matt Pocock's Skills Repo

David uses these. A composable set of Claude Code / Cursor skills built around the idea that software-engineering fundamentals matter more than ever. [github.com/mattpocock/skills](https://github.com/mattpocock/skills)

Notable skills:
- **grill-me** — Relentless interview about a plan until every decision branch is resolved, then a summary. [aihero.dev](https://www.aihero.dev/my-grill-me-skill-has-gone-viral)
- **grill-with-docs** — Evolution of grill-me for existing codebases: questioning backed by a `context.md` ubiquitous-language glossary and Architectural Decision Records (ADRs). [aihero.dev](https://www.aihero.dev/grill-with-docs)
- **tdd** — Red-green-refactor test-driven development loop.
- **to-prd** — Synthesizes conversation context into a formal PRD.
- **to-issues** — Converts a plan into independently-grabbable GitHub issues.
- **diagnose** — Structured debugging loop for hard bugs / performance issues.
- **handoff** — Compacts a conversation for agent-to-agent knowledge transfer.
- **zoom-out** — Requests broader system context for unfamiliar code.
- **caveman** — Compressed communication mode (~75% fewer tokens).
- **write-a-skill** — Scaffolds new skills with proper structure.
- **git-guardrails-claude-code** — Blocks dangerous git commands.

### Knowledge / RAG Alternative

- **LLM Wiki** (Karpathy pattern) — Model-maintained, interlinked markdown knowledge base instead of RAG; pre-compiles knowledge into entity pages so answers compound over time. [github.com/nashsu/llm_wiki](https://github.com/nashsu/llm_wiki)

### Terminals

- **cmux** — Native macOS terminal built for running AI agents in parallel (vertical tabs, per-tab git/PR/agent status). [overview](https://www.explainx.ai/blog/cmux-terminal-ai-coding-agents-2026)
- **Warp** — Agentic terminal / development environment with remote-agent support.

### Training / Experimentation

- **nanochat** — Karpathy's minimal full-stack ChatGPT pipeline. The "auto-regression / overnight experiments" tool referenced is his **autoresearch**, which lets an agent run training experiments autonomously overnight. [github.com/karpathy/autoresearch](https://github.com/karpathy/autoresearch)

### Inference

- **Fireworks AI** — Fast hosted inference (David runs Kimi here). [fireworks.ai](https://fireworks.ai)
