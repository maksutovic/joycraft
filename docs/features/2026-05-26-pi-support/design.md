# Pi Support — Design Discussion

> **Brief:** `docs/features/2026-05-26-pi-support/brief.md`
> **Research:** `docs/research/2026-05-26-pi-support.md`
> **Status:** Awaiting review
> **Date:** 2026-05-26

---

## Section 1: Current State

### init.ts (`src/init.ts`)
- `init(dir, opts)` unconditionally installs skills to `.claude/skills/` (18 Claude skill files) and `.agents/skills/` (18 Codex skill files)
- Generates `CLAUDE.md` via `generateCLAUDEMd()` and `AGENTS.md` via `generateAgentsMd()`
- Creates `.claude/settings.json` with permissions, SessionStart hook, PreToolUse hook
- Creates `.claude/hooks/joycraft-version-check.mjs` and `.claude/hooks/joycraft/block-dangerous.sh`
- Writes `.joycraft-version` with SHA256 hashes
- Validates `.gitignore` for `.claude/` exclusion

### detect.ts (`src/detect.ts`)
- `detectStack(dir)` returns `StackInfo` (language, packageManager, commands, framework, testingStrategy)
- Detects Node, Python, Rust, Go, Swift, Flutter, Xcode, Makefile, Dockerfile
- **Does not detect the coding agent** — no Pi, Claude Code, or Codex detection logic exists

### bundled-files.ts (auto-generated)
- Exports `SKILLS: Record<string, string>` (18 Claude skills as inline markdown)
- Exports `CODEX_SKILLS: Record<string, string>` (18 Codex skills — same filenames, different invocation syntax)
- Exports `TEMPLATES: Record<string, string>` (tree of template files)
- Generated by `scripts/generate-bundled-files.mjs` from `src/claude-skills/`, `src/codex-skills/`, `src/templates/`

### Skills (18 files, 2 variants each)
- All 18 skills use standard Agent Skills format (YAML frontmatter + markdown body)
- Claude variants: use `/joycraft-` prefix for invocations, can reference Claude-specific tools (`TodoWrite`, `EnterWorktree`, `LSP`)
- Codex variants: use `$joycraft-` prefix for invocations, no Claude-specific tools
- **Two skills use sub-agent spawning:**
  - `joycraft-research` — Phase 2: "Spawn a subagent to perform the research" (line 24)
  - `joycraft-verify` — Step 4: "Spawn a concurrent subagent thread" (line 82)
- **Pi compatibility gap:** Pi uses `/skill:joycraft-*` command format (not `/joycraft-*` or `$joycraft-*`). Pi has no sub-agent spawning API.

### upgrade.ts (`src/upgrade.ts`)
- Compares installed file hashes against `.joycraft-version`
- Auto-updates unmodified files, prompts for customized files
- Cleans up deprecated skill directories
- Runs forced migration from flat layout to per-feature layout
- **No Pi-specific upgrade path exists** — upgrade only knows `.claude/skills/` and `.agents/skills/`

### cli.ts (`src/cli.ts`)
- Three commands: `init`, `upgrade`, `init-autofix`
- Uses `commander` library with `--force`, `--yes` flags
- CLI entry point: `dist/cli.js` (bin: `joycraft`)

### Test infrastructure
- Vitest with 31 test files in `tests/`
- Fixture directories in `tests/fixtures/` for language detection
- `tests/init.test.ts` covers: skill creation, CLAUDE.md/AGENTS.md, Codex skills, settings, permissions, force/idempotency
- `tests/codex-skill-parity.test.ts` enforces parity between Claude and Codex skill variants

## Section 2: Desired End State

### `npx joycraft init` detects Pi
- When run from a Pi project, `init` discovers Pi and tailors installation
- A new `detectPi(dir)` function (or flag in `StackInfo`) identifies Pi projects
- Pi detection: presence of `.pi/` directory, or Pi's session files, or the `pi` binary

### Skills installed to Pi-compatible locations
- Skills go to `.pi/skills/joycraft-*/SKILL.md` (Pi's project-local discovery path)
- `.agents/skills/` still works too, but `.pi/skills/` is the Pi-idiomatic location
- Skill invocation uses `/skill:joycraft-*` — Pi's command format
- `joycraft-research` and `joycraft-verify` are adapted to use Pi's session API instead of sub-agents

### Bash scripts as the tool belt
- `joycraft-spec-status` — reads `docs/features/<slug>/specs/README.md`, counts completed specs
- `joycraft-mark-done` — updates a spec's `status:` from `active` to `Complete` in its YAML frontmatter
- `joycraft-next-spec` — finds the next uncompleted spec respecting dependency order, prints its path
- `joycraft-session-end` (bash version) — commits, captures discoveries, runs validation

### One minimal Pi extension (~40 lines)
- Registers a `joycraft_next_spec` tool callable by the LLM
- The tool runs bash scripts → marks current spec done → session-end → `ctx.newSession()` → sends `/skill:joycraft-implement <next-spec>` as kickoff message
- Lives at `.pi/extensions/joycraft-pipeline.ts`

### Autonomous pipeline flow
1. User starts with `/skill:joycraft-implement <first-spec>`
2. Implement skill runs TDD, completes spec
3. Implement skill calls `joycraft_next_spec` tool
4. Extension runs bash scripts: marks spec done, runs session-end
5. Extension calls `ctx.newSession()` with kickoff: `/skill:joycraft-implement <next-spec>`
6. New session starts with clean context, picks up next spec
7. Repeats until no more specs → reports pipeline complete

### Open-weight model support
- Skills are tested on DeepSeek v4 and Kimi K2.6
- Simplified skill variants created only where testing shows failures
- `joycraft-tune` gains a model capability check that optionally flags below-threshold models

## Section 3: Patterns to Follow

### Pattern 1: Unconditional dual install in init.ts
```typescript
// init.ts:72-98 — current pattern for skills installation
for (const [filename, content] of Object.entries(SKILLS)) {
  const skillName = filename.replace(/\.md$/, '');
  const skillDir = join(skillsDir, skillName);
  ensureDir(skillDir);
  writeFile(join(skillDir, 'SKILL.md'), content, opts.force, result);
}
```
**Follow for Pi:** Add a third loop for Pi skills. Pi skills use the Codex variant content (no Claude-specific tools) but with `/skill:joycraft-` invocation syntax. Install to `.pi/skills/<name>/SKILL.md`.

### Pattern 2: Detection chain in detect.ts
```typescript
// detect.ts:232-248 — detector chain, returns first match
const detectors = [
  detectNode, detectPython, detectRust, detectGo,
  detectFlutter, detectSwift, detectXcode,
  detectMakefile, detectDockerfile,
];
for (const detect of detectors) {
  const result = detect(dir);
  if (result) return result;
}
```
**Follow for Pi:** No new chain entry needed in `detectStack()`. Pi detection is separate from language stack detection. Add a standalone `detectAgent(dir)` function or detect Pi inline in `init()` by checking for `.pi/` directory.

### Pattern 3: `InitResult` tracking in init.ts
```typescript
// init.ts:27-36 — result tracking
interface InitResult {
  created: string[];
  skipped: string[];
  modified: string[];
  warnings: string[];
}
```
**Follow for Pi:** Reuse the same `InitResult` structure. Add Pi-specific entries to `.joycraft-version` hashes so `upgrade` can track them.

### Pattern 4: Dual CLAUDE.md / AGENTS.md generation
```typescript
// init.ts:111-131 — separate generation for each agent
const content = generateCLAUDEMd(projectName, stack, existingSkills);
const agentsContent = generateAgentsMd(projectName, stack);
```
**Follow for Pi:** AGENTS.md is already Pi-compatible. No new generation needed — Pi auto-discovers AGENTS.md from the project root. Do NOT generate a separate Pi-specific context file.

### Pattern 5: Upgrade with version hashing
```typescript
// upgrade.ts — managed file tracking
const managedFiles = getManagedFiles(); // SKILLS + TEMPLATES + CODEX_SKILLS
const installedHashes = versionInfo?.files ?? {};
// Compares hashes, auto-updates unmodified, prompts for customized
```
**Follow for Pi:** Add Pi skill files to `getManagedFiles()` return value and to the version hashes written during init. This makes `npx joycraft upgrade` handle Pi skill updates transparently.

### Pattern 6: Test fixture directories
```typescript
// tests/detect.test.ts — fixture-based detection tests
const fixtures = join(__dirname, 'fixtures');
it('detects npm with Next.js framework', async () => {
  const result = await detectStack(join(fixtures, 'node-npm'));
  expect(result.language).toBe('node');
});
```
**Follow for Pi:** Add a `pi-project` fixture directory with `.pi/` dir and `.pi/settings.json` to test Pi detection.

### Pattern 7: `writeFile` helper in init.ts
```typescript
// init.ts:38-43
function writeFile(path, content, force, result) {
  if (existsSync(path) && !force) { result.skipped.push(path); return; }
  writeFileSync(path, content, 'utf-8');
  result.created.push(path);
}
```
**Follow for Pi:** Use the same `writeFile` helper for Pi skill installation. Identical idempotency semantics.

### Pattern 8: `.json` settings merge in init.ts
```typescript
// init.ts:161-192 — settings.json merge
let settings = JSON.parse(readFileSync(settingsPath, 'utf-8'));
if (!settings.hooks) settings.hooks = {};
// ... append without duplicating
writeFileSync(settingsPath, JSON.stringify(settings, null, 2) + '\n', 'utf-8');
```
**Follow for Pi:** Pi uses `.pi/settings.json`. No need to merge settings on init for Pi — skills are auto-discovered from `.pi/skills/` directory presence. The bash scripts and extension don't require settings entries. However, we MAY optionally add `skills` paths to `.pi/settings.json` for cross-loading from `.claude/skills/` if the user wants backward compatibility.

## Section 4: Resolved Design Decisions

> **Decision:** No detection logic for Pi in `detectStack()`. Pi detection is done inline in `init()` by checking for `.pi/` directory existence.
> **Rationale:** `detectStack()`'s purpose is language stack detection (Node, Python, etc.), not agent detection. Adding agent detection to the same module conflates concerns. The check is trivial: `existsSync(join(dir, '.pi'))`. `init()` already does similar checks (e.g., for `.claude/skills/`, `.gitignore`).
> **Alternative rejected:** Adding `agent: 'claude' | 'codex' | 'pi'` to `StackInfo`. This would require modifying all StackInfo consumers and test fixtures. The detection is only needed in `init()`.

> **Decision:** Pi skills use the Codex skill content as their base, not the Claude content.
> **Rationale:** Claude skills contain Claude-specific tool references (`TodoWrite`, `EnterWorktree`, `LSP`) that don't apply to Pi. Codex skills are already cleaned of these. The invocation syntax difference (`/joycraft-` → `/skill:joycraft-`) is handled at install time by a string replace in `init()`, or by adding a third skill variant directory (`src/pi-skills/`).
> **Alternative rejected:** Shipping Claude skills as-is to Pi. Risk of broken tool references. Codex variants are the safer base.

> **Decision:** Add a third skill source directory (`src/pi-skills/`) for Pi-specific skill variants.
> **Rationale:** The invocation syntax `/joycraft-` vs `/skill:joycraft-` is a mechanical difference but affects every skill that references another skill or emits slash commands. A string replace at install time is fragile and hard to test. A third `src/pi-skills/` directory, generated alongside Claude and Codex variants, is explicit, testable, and follows the existing pattern (`src/claude-skills/`, `src/codex-skills/`). The `generate-bundled-files.mjs` script already reads flat `.md` directories — adding a third is a 2-line change. The bundled-files.ts export would be `PI_SKILLS`.
> **Alternative rejected:** Runtime string replacement. Would need to handle edge cases (code blocks, file paths containing "/joycraft-"). A third source is more maintainable.

> **Decision:** Adapted `research` and `verify` skills live in `src/pi-skills/` as Pi-specific variants.
> **Rationale:** The sub-agent → session API adaptation is a substantive rewrite of key skill phases, not a mechanical replacement. Keeping adapted versions alongside other Pi skills in `src/pi-skills/` makes the adaptation surface explicit.
> **Alternative rejected:** A single "universal" skill that detects the agent at runtime. This adds complexity to every skill invocation and breaks the progressive disclosure model.

> **Decision:** Bash scripts live in `.pi/scripts/joycraft/` with a `README.md` for progressive disclosure.
> **Rationale:** Follows Pi's philosophy: bash scripts + READMEs for the tool belt. The README is loaded by the agent on first `read` of the directory, providing progressive disclosure. Scripts are simple (read YAML frontmatter, find next spec, mark done) — each ~15-30 lines. No npm dependencies needed.
> **Alternative rejected:** Node.js scripts. Bash is simpler for file reading, string matching, and is Pi-idiomatic. `joycraft` CLI commands would require `npx joycraft` which adds dependency on npm registry availability.

> **Decision:** The pipeline extension goes in `.pi/extensions/joycraft-pipeline.ts` (project-local).
> **Rationale:** `npx joycraft init` writes it there. It's auto-discovered by Pi without any settings changes. Project-local means it's versioned with the project. The extension is ~40 lines — a single TypeScript file referencing `typebox` for the tool parameter schema.
> **Alternative rejected:** Global extension (`~/.pi/agent/extensions/`) or Pi package. Global means it's not project-versioned. Pi package adds distribution complexity (`pi install npm:joycraft`) that the brief explicitly defers.

> **Decision:** The `joycraft_next_spec` tool is the single extension-registered tool — not multiple tools.
> **Rationale:** One tool, one responsibility: transition to the next spec. It encapsulates the full handoff: mark-done → session-end → new-session → kickoff. Multiple tools would fragment the pipeline logic and require the LLM to orchestrate between them. The extension internally calls all three bash scripts.
> **Alternative rejected:** Registering `mark_done`, `session_end`, and `next_spec` as separate tools. Would require the LLM to sequence them correctly and handle errors between steps. Single tool = single atomic operation.

> **Decision:** `npx joycraft upgrade` treats Pi skills like any other managed files — hash comparison, auto-update for unmodified, prompt for customized.
> **Rationale:** The upgrade infrastructure (version hashing, `getManagedFiles()`) already handles this. Adding `PI_SKILLS` to the managed file set makes upgrades transparent. No new upgrade logic needed.
> **Alternative rejected:** Separate `joycraft upgrade --pi` command. Unnecessary fragmentation. Pi skills are just another set of managed files.

> **Decision:** Pi skills are always installed during init, not conditionally.
> **Rationale:** The existing pattern is unconditional install to both `.claude/skills/` and `.agents/skills/`. The `.pi/skills/` directory is Pi-specific but harmless on non-Pi projects (it's just unused markdown files). This avoids conditional complexity. The init summary can note "Pi skills installed (use in Pi with /skill:joycraft-*)" regardless of detected agent.
> **Alternative rejected:** Conditional install only when `.pi/` is detected. This means re-running init after adopting Pi. Unconditional install mirrors the existing dual-install pattern and is simpler.

> **Decision: AGENTS.md remains a standalone file — no Pi-specific variant. Remove the "See CLAUDE.md" cross-reference from `generateAgentsMd()`.**
> **Rationale:** Both CLAUDE.md and AGENTS.md are already compatible with their respective harnesses. The only issue is the line `> Auto-generated by Joycraft. See CLAUDE.md for detailed instructions.` which makes AGENTS.md a second-class citizen in Pi-only projects. Removing that one line makes AGENTS.md self-contained. No third file generation needed — Pi auto-discovers AGENTS.md natively.
> **Alternative rejected:** Pi-specific AGENTS.md generator. Unnecessary duplication. Both files should be standalone and work for their respective harnesses without cross-references.

> **Decision: Spec queue uses a hybrid approach — JSON manifest (`.joycraft-spec-queue.json`) as the authoritative machine-readable queue, README.md as the human-readable view. Both generated by `joycraft-decompose`.**
> **Rationale:** JSON is deterministic for tool calling (bash scripts and the TypeScript extension can parse it reliably). README.md remains for human navigation. The JSON is authoritative — if they drift, the JSON wins. The decompose skill generates both from the same data, keeping them in sync. The manifest format:
> ```json
> {
>   "feature": "2026-05-26-pi-support",
>   "specs": [
>     { "id": 1, "file": "add-pi-detection.md", "depends_on": [], "status": "active" },
>     { "id": 2, "file": "add-pi-skills.md", "depends_on": [1], "status": "active" }
>   ]
> }
> ```
> **Alternative rejected:** Parse markdown tables in bash (fragile). Scan YAML frontmatter only (dependencies are in README, not spec frontmatter). Single JSON without README (loses human readability).

> **Decision: Research and verify use Pi's subagent pattern — spawn separate `pi` processes with isolated context.**
> **Rationale:** Pi ships a subagent extension (`examples/extensions/subagent/`) that spawns separate `pi` subprocesses with clean context windows, streaming output, and abort support. This is architecturally equivalent to Claude Code's sub-agent spawning. Joycraft defines "researcher" and "verifier" agent definitions (`.pi/agents/joycraft-researcher.md`, `.pi/agents/joycraft-verifier.md`) with `tools: read, grep, find, ls, bash`. The pipeline extension registers a `subagent` tool (or bundles a minimal subagent spawner). The adapted skills instruct the agent: "Use the subagent tool with agent `joycraft-researcher` and task: [questions]." True context separation — the verifier cannot see implementation context.
> **Alternative rejected:** Session API (`ctx.newSession()`) — no way to read back results programmatically. Inline role-switching — no context separation, degrades verification quality. Hybrid — loses the context-separation benefit for verify.

> **Decision: No automatic model capability check in v1. Test-first approach.**
> **Rationale:** The brief states: "Don't pre-optimize for weaker models without data. Test first, then tune." Adding a capability check without evidence of failure is premature optimization. If testing reveals specific models fail on specific skills, add targeted checks (or simplified skill variants) at that point. The `joycraft-tune` skill can gain this capability in a future iteration if needed.
> **Alternative rejected:** Hardcoded weak-model list (goes stale). Experimental benchmark (complex, adds latency to `tune`). Both add complexity without data proving they're needed.

> **Decision: Error recovery uses pause-and-flag — `joycraft_next_spec` detects failure, marks spec `blocked` in the JSON manifest, and exits without transitioning.**
> **Rationale:** Safest approach. No cascading failures. The pipeline stops and the human investigates. A spec marked `blocked` is skipped by subsequent `joycraft-next-spec` invocations until the human unblocks it. Detecting failure: the `joycraft_next_spec` tool can check exit codes from bash scripts, or the spec's `status` field if the implement skill updates it on failure.
> **Alternative rejected:** Skip-and-continue (complex dependency resolution in bash). Retry N times (wastes tokens on unfixable issues). Both add complexity for v1.

> **Decision: Pi skills live in a third source directory (`src/pi-skills/`) alongside `src/claude-skills/` and `src/codex-skills/`. Each variant independently editable.**
> **Rationale:** The research and verify skills need substantive rewrites to use the subagent pattern — not mechanical substitutions. A third directory follows the existing pattern, makes differences explicit, and extends the parity test to 3-way comparison. The `generate-bundled-files.mjs` change is 2 lines (`PI_SKILLS_DIR`). The bundled export is `PI_SKILLS`.
> **Alternative rejected:** Derive from Codex at build time. The transformation logic for sub-agent section rewrites is fragile and hard to test. A third directory is more maintainable.
