## What

`npx joycraft init` gains Pi detection and full Pi installation. When `init()` runs, it checks for a `.pi/` directory and reports Pi as a detected agent in the summary. Regardless of detection, it unconditionally installs Pi skills to `.pi/skills/<name>/SKILL.md`, pipeline scripts to `.pi/scripts/joycraft/`, the pipeline extension to `.pi/extensions/joycraft-pipeline.ts`, and subagent definitions to `.pi/agents/`. Pi managed files are added to version hashing. The AGENTS.md generator is fixed to remove the "See CLAUDE.md" cross-reference.

## Why

Without init wiring, there's no path from `npx joycraft init` to a working Pi setup — the skills, scripts, extension, and subagent defs would all exist in source but never land in a user's project.

## Acceptance Criteria

- [ ] `init()` detects `.pi/` directory and reports Pi in the summary output
- [ ] Pi skills from `PI_SKILLS` are installed to `.pi/skills/<name>/SKILL.md`
- [ ] Pipeline bash scripts are installed to `.pi/scripts/joycraft/` with executable permissions
- [ ] Pipeline README is installed to `.pi/scripts/joycraft/README.md`
- [ ] Pipeline extension is installed to `.pi/extensions/joycraft-pipeline.ts`
- [ ] Subagent definitions are installed to `.pi/agents/joycraft-researcher.md` and `.pi/agents/joycraft-verifier.md`
- [ ] Pi managed files appear in `.joycraft-version` hashes
- [ ] `generateAgentsMd()` no longer emits "See CLAUDE.md for detailed instructions"
- [ ] Init summary mentions Pi support regardless of detection
- [ ] Idempotency: running init twice does not duplicate or error (existing files skipped unless `--force`)
- [ ] Build passes (`pnpm build`)

## Constraints

- MUST: Pi skills are installed unconditionally — same pattern as Claude/Codex dual install (harmless unused `.md` files on non-Pi projects)
- MUST: Detection is inline in `init()` (check `existsSync(join(dir, '.pi'))`) — no changes to `detectStack()`
- MUST: Pipeline scripts must have executable permissions (`chmod 755`) when installed
- MUST: Reuse existing `writeFile` helper and `InitResult` tracking — no new patterns
- MUST: AGENTS.md fix is a one-line removal: delete `> Auto-generated by Joycraft. See CLAUDE.md for detailed instructions.` from `generateAgentsMd()`
- MUST NOT: Generate a Pi-specific context file — AGENTS.md is already Pi-compatible
- MUST NOT: Modify `.pi/settings.json` — skills are auto-discovered from `.pi/skills/` directory presence
- MUST NOT: Add Pi detection to `detect.ts` — agent detection is separate from language stack detection

## Approach

### Init flow changes

After the existing Codex skill install loop (after line ~109), add:

```
// Pi skills — install unconditionally (same pattern as Claude/Codex)
const piSkillsDir = join(targetDir, '.pi', 'skills');
for (const [filename, content] of Object.entries(PI_SKILLS)) {
  const skillName = filename.replace(/\.md$/, '');
  ensureDir(join(piSkillsDir, skillName));
  writeFile(join(piSkillsDir, skillName, 'SKILL.md'), content, opts.force, result);
}

// Pi pipeline runtime
const piScriptsDir = join(targetDir, '.pi', 'scripts', 'joycraft');
ensureDir(piScriptsDir);
writeFile(join(piScriptsDir, 'joycraft-spec-status'), PI_SCRIPTS['joycraft-spec-status'], opts.force, result);
// ... etc

const piExtDir = join(targetDir, '.pi', 'extensions');
ensureDir(piExtDir);
writeFile(join(piExtDir, 'joycraft-pipeline.ts'), PI_EXTENSIONS['joycraft-pipeline.ts'], opts.force, result);

const piAgentsDir = join(targetDir, '.pi', 'agents');
ensureDir(piAgentsDir);
writeFile(join(piAgentsDir, 'joycraft-researcher.md'), PI_AGENTS['joycraft-researcher.md'], opts.force, result);
// ... etc
```

### Detection and summary

At the top of `init()`:
```typescript
const isPi = existsSync(join(targetDir, '.pi'));
```

In `printSummary()`:
```
if (isPi) console.log('  Detected agent: Pi');
```

Always show in next steps:
```
console.log('    Pi: Skills installed to .pi/skills/. Use /skill:joycraft-* to invoke.');
```

### AGENTS.md fix

In `generateAgentsMd()`, change:
```
'> Auto-generated by Joycraft. See CLAUDE.md for detailed instructions.',
```
to:
```
'> Auto-generated by Joycraft.',
```

**Rejected alternative:** Conditional Pi install (only when `.pi/` detected). Means users who adopt Pi later must re-run init. Unconditional install mirrors the existing dual-install pattern and is simpler.

## Edge Cases

| Scenario | Expected Behavior |
|----------|------------------|
| `.pi/skills/` already has non-Joycraft skills | Joycraft skills are added alongside — no conflict (Pi discovers all directories) |
| `.pi/extensions/` already has other extensions | `joycraft-pipeline.ts` is added — Pi auto-discovers all `.ts` files |
| `.pi/` doesn't exist (not a Pi project) | Created by init — harmless directory with skill files not discovered by any agent |
| `--force` flag passed | All Pi files overwritten regardless of existing content |
| AGENTS.md was already customized by user (remove line beforehand) | Init skips AGENTS.md if it exists and no `--force` — user's custom content is preserved |
| Manifest scripts reference missing template | Init errors at install time with clear message — this is a build-time invariant |
