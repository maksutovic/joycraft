# Design Discussion: Context Layer + Onboarding + Bugfix Rename

> Sits between the brief and decomposition. Captures my understanding of *how* this lands in the codebase so you can catch wrong assumptions before specs get written. No research doc was provided — Section 1 is from direct codebase exploration.

The brief is unusually well-resolved (8 decisions already answered). This design therefore focuses on the *implementation shape* — where each change lands, which existing pattern it follows, and the few places where the brief's intent meets a codebase reality that forces a real choice.

## Section 1: Current State

### Skills (Claude + Codex mirrors)
- Claude skills: `src/claude-skills/*.md` (15 files), installed to `.claude/skills/<name>/SKILL.md` (`init.ts:67-73`).
- Codex skills: `src/codex-skills/*.md` (15 mirror files), installed to `.agents/skills/<name>/SKILL.md` (`init.ts:75-95`).
- Frontmatter drives auto-invocation. Claude skills carry 3 fields (`name`, `description`, `instructions: N`); Codex mirrors carry 2 (no `instructions`). Body differs only at platform-specific lines (`.claude/` vs `.agents/`, `.claude/settings.json` vs "deny patterns configuration").
- `joycraft-add-fact.md` (38 instructions) routes a one-liner into one of 5 flat fact-docs by signal-word matching (`src/claude-skills/joycraft-add-fact.md:19-50`). Context docs it touches carry a 2-field shared frontmatter (`last_updated`, `last_updated_by`) — no single owner.
- `joycraft-tune.md` is the leanest skill (15 instructions). Its Step 3 scores **7 dimensions** including a `Documentation` row and a `Knowledge Capture` row that already reads `docs/context/*.md` (`joycraft-tune.md:24-32`). Its Step 5 risk interview "Skip[s] if `docs/context/` already has content" — all-or-nothing (`joycraft-tune.md:49`).
- `joycraft-bugfix.md` is already migrated: writes to `docs/bugfixes/<area>/`, uses `area:` frontmatter, maintains a per-area `README.md` index. (Part A skills are done.)

### CLI
- `init.ts:48` creates exactly one managed docs subdir: `docs/context/`. Everything else is lazy-created by skills. No `docs/context/reference/` today.
- `init.ts:290-299` prints next-steps; line 296 leads with `/joycraft-new-feature` (wrong for a no-harness newcomer).
- `improve-claude-md.ts` parses CLAUDE.md into `## ` (level-2) sections (`parseSections`, lines 14-38), checks each via `hasSection(sections, /regex/i)`, and appends missing sections at the end (`additions.join('\n\n')`, lines 225-226). It generates a "Getting Started with Joycraft" section whose table leads with `/joycraft-tune` then `/joycraft-new-feature` (lines 110-125). No `## Context Map` section exists or is generated.
- `migration.ts` plans flat→per-feature moves. Spec subdirs that match a brief slug get moved into `docs/features/<slug>/specs/`; **subdirs with no matching brief become `orphans.specsDirs`** (`migration.ts:148-149`) — i.e., the bugfix areas. `upgrade.ts:134-140` currently prints them under "Left in place — area-level specs (e.g., bugfix areas)" and does **not** move them.
- `runForcedMigration` (`upgrade.ts:152-179`) prints a summary then applies — **no y/N gate** (the `askUser` helper at line 100 is only used later for customized-file overwrites). Migration runs unconditionally during upgrade.
- `src/bundled-files.ts` is `@generated` — three Records (`SKILLS`, `TEMPLATES`, `CODEX_SKILLS`). Regenerated by `node scripts/generate-bundled-files.mjs`: skills read flat, templates read recursively (so `src/templates/context/reference/foo.md` → key `context/reference/foo.md` automatically).
- `src/templates/context/` holds the 5 fact-doc templates today; no `reference/` subdir.

### Two template dirs (gotcha)
- `src/templates/` — **bundled**, copied into user projects. The context layer's new reference templates go here.
- `templates/` (repo root) — development-reference only, NOT bundled.

## Section 2: Desired End State

**New skills** (each as a Claude + Codex pair):
- `joycraft-add-context.md` — scaffold one long-form reference doc into `docs/context/reference/<slug>.md` from a template; create-or-update the `## Context Map` pointer row in CLAUDE.md idempotently.
- `joycraft-gather-context.md` — read existing docs (README + `docs/` + CLAUDE.md), summarize coverage, offer a *gap-only* optional interview, route facts→`add-fact` buckets and reference→`add-context` scaffolding, wire every doc into `## Context Map`.
- `joycraft-setup.md` — one-screen alias carrying newcomer vocabulary in `description`, body routes to `/joycraft-tune`.

**New bundled templates** under `src/templates/context/reference/`: `design-system.md`, `frontend-methodology.md`, `backend.md`, `testing.md`, `reference-doc.md` (generic fallback).

**Modified skills:**
- `joycraft-tune.md` — Step 1/3 recognize the context layer (incl. `docs/context/reference/`); Step 5 invokes `/joycraft-gather-context` on first run instead of the narrow risk-only interview; Documentation dimension rewards lean+pointered CLAUDE.md and flags monoliths >~200 lines.
- The 5 skills with generic `docs/specs/my-feature/...` placeholders (`verify`, `implement`, `tune`, `implement-level5`, `collaborative-setup`) updated to `docs/features/<slug>/specs/...`.

**Modified CLI:**
- `improve-claude-md.ts` — generates an idempotent `## Context Map` stub section + leads the Getting Started table with `/joycraft-setup`.
- `init.ts` — next-steps lead with `/joycraft-setup`; `docs/context/reference/` produced (directly or via first skill use).
- `migration.ts` + `upgrade.ts` — orphan spec dirs become a *planned bugfix move* `docs/specs/<area>/` → `docs/bugfixes/<area>/`, behind a preview + confirmation gate, never clobbering an existing target.

**Regenerated:** `src/bundled-files.ts`; `pnpm test --run && pnpm typecheck` green.

## Section 3: Patterns to Follow

### Idempotent CLAUDE.md section (for `## Context Map`)
Follow the exact `generate…Section()` + `hasSection()` + push pattern (`improve-claude-md.ts:173-227`):
```ts
if (!hasSection(sections, /context\s*map/i)) {
  additions.push(generateContextMapSection());
}
```
The stub section header should carry the lean-CLAUDE.md teaching (Decision 5): a one-line "Keep this file lean — link out, don't inline." The skills (`add-context`/`gather-context`) maintain the rows afterward by string-editing the live file.

### Skill frontmatter & invocation vocabulary
Match the existing description grammar — *action verb + specific artifact + outcome* (`add-fact.md:1-5`, `tune.md:1-5`). `joycraft-setup`'s description is the entire point of the alias: it must carry newcomer words ("set up / get started / first time / configure my project"). `gather-context` carries onboarding words distinct from `add-fact`'s operational signal-words, so they don't cross-trigger.

### Codex mirror discipline
For every new/changed Claude skill, produce the Codex twin: drop the `instructions:` field, swap `.claude/`→`.agents/` and `.claude/settings.json`→"deny patterns configuration" in the body. Otherwise identical (compare `claude-skills/joycraft-tune.md:13,51` vs `codex-skills/joycraft-tune.md:12,50`).

### Self-contained skills (gotcha #3)
`gather-context` cannot import `add-fact`'s routing — it must inline the minimal "could this be one table row? → fact; does it take paragraphs? → reference" test (Decision 6). `add-fact` and `add-context` stay the standalone single-doc primitives.

### Templates auto-bundle
Drop reference templates into `src/templates/context/reference/`; `readTreeDir` picks them up recursively (`generate-bundled-files.mjs:38-47`). Match the existing context-template shape: H1 title, `>` blockquote purpose, italic example rows the user deletes (`src/templates/context/production-map.md`).

### Plan-then-apply migration
The bugfix move follows the existing `Move`/`MigrationPlan`/`applyMigration` structure (`migration.ts:16-35,177-200`), including the `EXDEV` cross-device fallback in `moveFsItem` (lines 162-175) and the skip-if-target-exists guard (line 183).

## Section 4: Resolved Design Decisions

> **Decision:** `## Context Map` stub renders only the header + teaching line + an empty table skeleton when there are no docs to point at; skills fill rows later.
> **Rationale:** Mirrors how `improve-claude-md.ts` emits self-contained stub sections, and keeps the generated file honest (no fake pointer rows). The header teaching line satisfies Decision 5's "taught in both places."
> **Alternative rejected:** Pre-populating rows for the 5 fact-docs at init — they may not exist yet, producing dangling pointers and contradicting "link only what's real."

> **Decision:** `joycraft-setup` is a thin router with no logic of its own — its body is essentially "this is the first-run door; run `/joycraft-tune`," and tune does the work (including invoking `gather-context` on first run).
> **Rationale:** Brief Decision (Part D) + zero-duplication. The whole value is the `description` field existing in newcomer vocabulary; the body must not fork tune's logic.
> **Alternative rejected:** `setup` reimplementing detect+route — duplicates tune, drifts over time.

> **Decision:** `gather-context`'s existing-doc scan is README + `docs/**` + CLAUDE.md only; a deeper code-inference review is offered only if the user explicitly asks, with a cost note.
> **Rationale:** Brief Decision 8 — give agency, don't burn tokens by default.
> **Alternative rejected:** Auto code-inference scan in v1 (out of scope).

> **Decision:** Reference templates ship as 5 files (`design-system`, `frontend-methodology`, `backend`, `testing`, `reference-doc`), each H1 + purpose blockquote + section skeleton with deletable example prose.
> **Rationale:** Brief Decision 2; matches the established fact-doc template shape.
> **Alternative rejected:** A single mega-template with conditional sections — harder to invoke specifically, less useful as a starting skeleton.

> **Decision (was Q1 → B):** The `## Context Map` stub is generated in BOTH `init`'s fresh CLAUDE.md generation and `improve-claude-md.ts` merge path.
> **Rationale:** The section is core to the feature's thesis (lean CLAUDE.md, link out) and should greet a first-timer from minute one. Both paths reuse the same `generateContextMapSection()` helper to stay in sync.
> **Alternative rejected:** merge-path only — a brand-new project wouldn't see the section until first upgrade/tune, undercutting the onboarding goal.

> **Decision (was Q2 → force, no gate):** The bugfix move is FORCED, like the existing feature migration — no interactive y/N gate. It prints a clear "Migrating bugfix areas:" preview, applies, and still honors the "never clobber an existing `docs/bugfixes/<area>/`" skip guard. Orphan spec dirs stop being "left in place" and become planned moves.
> **Rationale:** User has a small, known userbase ("all my friends") and is fine telling them what changed; forcing keeps one uniform migration UX and avoids stdin coupling. The skip-if-target-exists guard (`migration.ts:183`) and the >50%-failure abort (`upgrade.ts:163`) provide the safety Decision 4 cares about without a prompt.
> **Alternative rejected:** A y/N gate scoped to bugfix moves — mixed UX, extra stdin handling, unnecessary for this userbase. (This relaxes brief Decision 4's "require the confirmation gate" by explicit owner call.)

> **Decision (was Q3 → A, clarified):** Codex parity means SAME CONTENT, Codex-idiomatic invocation. The only mechanical difference is the command sigil: Claude uses `/joycraft-…`, Codex uses `$joycraft-…` (confirmed in `src/codex-skills/*` — e.g. `joycraft-decompose.md:14` `$joycraft-new-feature`), plus the existing `.claude/`→`.agents/` and `.claude/settings.json`→"deny patterns configuration" swaps. So `setup`'s Codex mirror routes to `$joycraft-tune` with otherwise-identical body.
> **Rationale:** Owner clarified "parity" = identical logic, inferred into real Codex syntax — not literal `/` text.
> **Alternative rejected:** Reworded/abstracted Codex bodies — drifts from the Claude twin.

> **Decision (was Q4 → split by skill):** `add-context` writes immediately per-doc (incremental, option A). `gather-context` collects all gap answers and writes everything + the Context Map in ONE batch with a final confirm (option B).
> **Rationale:** Owner intent — `add-context` is a single-doc primitive (write-as-you-go fits); `gather-context` is a "do this in one go" guided pass, so one clean reviewable diff at the end is the right shape.
> **Alternative rejected:** Uniform per-answer writes for both — wrong for the batch-oriented onboarding pass.

> **Decision (was Q5 → A):** `docs/context/reference/` is lazy-created by `add-context`/`gather-context` on first write; no extra `ensureDir` in `init.ts`.
> **Rationale:** Honors init.ts's documented "solo-first: no preemptive ceremony" philosophy (`init.ts:~45`); empty dirs don't survive git anyway.
> **Alternative rejected:** Preemptive `ensureDir` in init — contradicts the stated design note.
